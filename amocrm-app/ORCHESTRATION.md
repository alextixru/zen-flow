# Оркестрация ночного цикла amocrm-app: роутинг моделей + правки промпта

Для будущей копии `ralph.sh` под этот цикл (сам `ralph/ralph.sh` НЕ трогать — он обслуживает piece-цикл).
План: `amocrm-app/PRD.md` (19 W-задач + 4 V-чекпоинта). Принцип квоты: **fable дорогой — минимум и только там, где цена ошибки архитектурная/безопасностная; sonnet дешёвый — вся механика по плотной спеке; opus — интеграционные развилки.**

## 1. Роутинг: задача → модель

| Задача | Модель | Почему |
|---|---|---|
| W001 скелет виджета + установка | **opus** | Спека плотная, но первая живая загрузка zip через браузер — неизведанные грабли UI amo; провал W001 блокирует весь проект. |
| W002 живая проверка CSP/размеров | **opus** | Исследовательская задача: интерпретация консоли/Network, фиксация фактов, от которых зависит фаза 1. Кода мало, суждений много. |
| W003 embed-handshake | **fable** | Архитектурный узел PoC: подъём EE-форка, signing key, JWT, туннели, `dev-stand.sh` — фундамент всех последующих живых проверок. |
| W004 UI-QA piece | **sonnet** | Прокликивание по готовому чек-листу, фиксация багов списком. Ничего не проектируется. |
| V001 чекпоинт PoC | **fable** | Валидатор (общая политика V-задач). |
| W005 каркас моста | **sonnet** | Полностью механическая сборка по спеке: файлы, env, таблица, /health названы поимённо. |
| W006 install-flow | **opus** | Идемпотентность, валидация входа, живое поведение `onSave`/return в amo — реальные развилки. |
| W007 /embed-token | **fable** | Выпуск токенов доступа к чужим проектам — security-вес: связка, claims, rate-limit; ошибка = дыра в managed-аутентификации. |
| W008 автодобавление origin | **fable** | Правка CSP-списка боевого форка + разбор SERVICE-принципала по коду; ошибка = либо сломанный embedding, либо расширенная поверхность атаки. |
| W009 автосоздание connection | **opus** | Обмен JWT→token→app-connections, сверка формы тела живым вызовом, идемпотентность по externalId. |
| W010 виджет v1 advanced_settings | **opus** | UI-развилки (4 состояния + ошибки сети), детект тёмной темы живьём, повторные configure(). |
| W011 lcard-кнопка | **sonnet** | Кнопка + переход по зафиксированному в W001 URL. Тривиально при готовом W010. |
| W012 ALLOWED-каталог | **sonnet** | Скрипт тегирования по готовому списку; разведка `pieceTagService` — grep по коду, не проектирование; есть escape «руками через UI». |
| V002 чекпоинт MVP | **fable** | Валидатор + первый секрет-скан блока с ключами/токенами. |
| W013 белый бэкдор + статика | **opus** | Разнос кода на загрузчик/приложение, контракт loader↔app — лёгкое проектирование, живой двухрежимный тест. |
| W014 /flows + DP-приёмник | **opus** | Интеграционная логика: обмен токенов с кэшем, проверка принадлежности flow, async-запуск. Безопасностный вес есть, но модель угроз уже прописана в спеке — fable не нужен. |
| W015 DP-блок манифеста + dpSettings | **opus** | Реверс механики записи поля из боевого кода Triggeron — главный неочевидный момент, sonnet застрянет. |
| W016 salesbot-шаг | **opus** | Открытый вопрос формата JSON: восстановление из reference/bpmn + живые пробы + escape-развилка. |
| W017 дедуп + очередь | **sonnet** | Спека даёт схему целиком (таблицы, hash-ключ, interval, лимиты); остаётся аккуратно закодить + тесты на моке. |
| V003 чекпоинт автоворонки | **fable** | Валидатор. |
| W018 card_sdk-вкладка | **opus** | Риск «вкладка заточена под товарные Promise» + fallback-развилка; два новых эндпоинта. |
| W019 mobile.frame_url | **sonnet** | Статическая страница + логирование контракта; безопасный дефолт («данных не показывать») жёстко прописан в спеке — суждений не требуется. |
| V004 финальный чекпоинт | **fable** | Валидатор + секрет-скан всей ветки + README. |

Итого: **sonnet — 6** (W004, W005, W011, W012, W017, W019), **opus — 10** (W001, W002, W006, W009, W010, W013, W014, W015, W016, W018), **fable — 7** (W003, W007, W008, V001–V004).

Границы между уровнями (если задача будет пересечена/добавлена): формы данных и файлы названы поимённо, суждений нет → sonnet; есть живая развилка/реверс чужого кода/неснятый вопрос → opus; ошибка стоит секретов, чужих данных или архитектуры моста → fable.

## 2. Фрагмент для копии ralph.sh

Заменяет блок `FABLE_TASKS_RE`/`pick_model()` (в piece-цикле — строки ~73–92). Отличия: путь к PRD, regex `W###`, три ветки.

```bash
PRD_FILE="amocrm-app/PRD.md"
# Трёхуровневый роутинг: sonnet — механика по плотной спеке, fable — V-чекпоинты и
# security/архитектурные узлы (handshake, embed-token, origins), остальное — opus.
SONNET_TASKS_RE='^(W004|W005|W011|W012|W017|W019)$'
FABLE_TASKS_RE='^(V[0-9]+|W003|W007|W008)$'
pick_model() {
  local task
  task=$(grep -E '^### - \[ \] ' "$PRD_FILE" | grep -v 'BLOCKED' | head -1 \
    | grep -oE '\b[WV][0-9]{3}[a-z]?\b' | head -1)
  if [ -z "$task" ]; then echo "fable"; return; fi   # план пуст/нечитаем — финалить пусть fable
  if echo "$task" | grep -qE "$FABLE_TASKS_RE";  then echo "fable";  return; fi
  if echo "$task" | grep -qE "$SONNET_TASKS_RE"; then echo "sonnet"; return; fi
  echo "opus"
}
```

Прочие правки копии: `BRANCH="feature/amocrm-app"`, `ACTIVITY="amocrm-app/activity.md"`, `LOGDIR="amocrm-app/logs"`, промпт — `claude -p "$(cat amocrm-app/PROMPT.md)"`.

**Проверить первым тиком:** алиасы `fable`/`opus` в `claude -p --model` подтверждены piece-циклом; `sonnet` — стандартный алиас (Sonnet 5), но его приёмистость этим CLI живьём не проверялась — если первый sonnet-запуск упадёт на выборе модели, заменить в `SONNET_TASKS_RE`-ветке echo на `opus` и записать факт в activity.

## 2а. Защита от runaway-loop (обязательна — разбор бага piece-цикла 2026-07-11)

Причина инцидента той ночи: session-limit (`You've hit your session limit · resets 7:40am`) не был распознан `sleep_if_limited`, цикл не ушёл в сон и сжёг 27 итераций подряд по 2–4с каждая. Два корня: (A) формулировка лимита плавает — regex её не знал; (B) фикс закоммитили в уже РАБОТАЮЩИЙ процесс, а bash парсит функции один раз при старте — живой процесс правку не подхватил. Копия оркестратора для amocrm-app ОБЯЗАНА иметь три страховки (любая ловит инцидент после 2-й итерации, а не 27-й):

1. **Эвристика длительности/размера — главный предохранитель, не зависит от текста.** Замерять длительность тика (`SECONDS` до/после `claude -p`) ИЛИ размер лога. Тик <30с или лог <200 байт = почти наверняка провал, не работа → сон (как при лимите). Это ловит ЛЮБУЮ будущую формулировку лимита/краша, которую regex не знает.
   ```bash
   t0=$SECONDS; claude -p "$(cat "$PROMPT_FILE")" --model "$MODEL" --dangerously-skip-permissions >"$log" 2>&1; dt=$((SECONDS-t0))
   if [ "$dt" -lt 30 ] || [ "$(wc -c <"$log")" -lt 200 ]; then
     fast=$((fast+1)); echo "[$(date '+%H:%M')] подозрительно быстрый тик (${dt}s) — возможно лимит/краш"
     sleep_if_limited "$log" || sleep 900   # распознали формулировку → спим до сброса; нет → 15м бэкофф
     [ "$fast" -ge 3 ] && { echo "3 быстрых тика подряд — стоп, нужен человек"; break; }  # circuit breaker
     continue
   fi
   fast=0
   ```
2. **Circuit breaker** (в фрагменте выше, счётчик `fast`): 3 мгновенных тика подряд → выход из цикла, а не докрутка бюджета вхолостую.
3. **Детекцию лимита — во внешний скрипт `amocrm-app/check-limit.sh`, вызываемый каждый тик** (`bash amocrm-app/check-limit.sh "$log"`), а НЕ функцией в теле цикла. Тогда правка паттернов подхватывается на следующем тике (свежий процесс bash), даже если оркестратор запущен давно — прямая митигация корня B. Внутри — всеядный `LIMIT_MSG_RE='hit your (session|usage) limit|usage limit reached|rate.?limit|limit.*reset'` + парсинг времени сброса.

Правило релиза: **правку живого оркестратора нельзя применить на ходу** — либо перезапускать цикл после правки ralph.sh, либо (для check-limit.sh) полагаться на то, что он перечитывается каждый тик.

## 3. Правки модели исполнения (эталон `ralph/PROMPT.md` → будущий `amocrm-app/PROMPT.md`)

Что переносится как есть: одна итерация = одна задача; BLOCKED/STUCK/partial-протокол; заземление отчёта; «хватает информации — действуй»; формат финального сообщения; правило субагентов «model всегда указан, дефолт sonnet».

Что менять (по разделам эталона):

1. **Границы.** Разрешённые пути: `amocrm-app/widget/**`, `amocrm-app/bridge/**`, `amocrm-app/PRD.md` (свой чекбокс/BLOCKED), `amocrm-app/activity.md` (своя запись); локально вне git — `.env.dev`, `bridge/.env`, `bridge/data/`. Явный запрет: `packages/**` (включая piece amocrm — у него СВОЙ цикл ralph), `ralph/**` (журнал соседнего цикла читать можно — вердикт T038 влияет на DP-механики, — писать нельзя), конфиги, `.claude/**`. Упёрся в правку форка → BLOCKED, не самодеятельность.
2. **Ветка.** `feature/amocrm-app` (создаётся от `feature/amocrm-piece` в W001, если ещё нет); коммиты `feat(amo-app): w0XX …` (ID строчными — commitlint). Соседний цикл коммитит в `feature/amocrm-piece` — никогда не переключаться на неё и не мержить.
3. **Документация перед кодом.** Вместо `.agents/features/*`: справочник в шапке PRD (обязателен всегда) + `research/embed-sdk.md` для задач моста/JWT, `research/amo-surfaces.md` для задач манифеста/locations, `reference/triggeron|bpmn/widget/` для DP/salesbot. Референсы заново не исследовать — спеки самодостаточны.
4. **Два стенда одновременно.** Секция «Стенды» вместо одной про amo: (а) amo `dzenteamdev.amocrm.ru` — как в эталоне; (б) форк EE (Postgres + API 8080 + worker 8082 + туннели). Правило: стенд не переживает итерацию — живые задачи начинают с `bridge/scripts/dev-stand.sh`; фоновые процессы запускать `nohup`-ом с логами в `/tmp`, НЕ оставлять обещаний «стенд поднят» без проверки `curl`.
5. **Браузерная секция (новая).** Живые UI-проверки — через chrome-devtools MCP или скилл `agent-browser`: подключиться к браузеру с живой сессией dzenteamdev; загрузка zip — руками агента через UI (путь фиксируется в W001); CSP-проверки — читать console + network; скриншоты класть рядом с логами, факты — в activity. Сессия разлогинена → «непроверено живьём» + добор на ближайшей V-задаче (для W001 — BLOCKED). Пароли не подбирать, 2FA не обходить.
6. **Секрет-гигиена (усилить относительно эталона).** Список секретов шире: приватный PEM, embed-JWT, amo-токены клиентов, install-ключи, `FORK_API_KEY`, админ-пароль превью. Перед КАЖДЫМ коммитом: `git diff | grep -cE 'eyJ0|BEGIN (RSA )?PRIVATE KEY'` = 0 и в `git status` нет `.env*`/`data/`/`*.pem`. В activity секреты не цитировать (только имена переменных).
7. **Verify.** Вместо turbo/lint-dev монорепо: `npx tsc --noEmit` + `npx vitest run` в `bridge/`, `./build.sh` виджета, curl-smoke эндпоинтов задачи, живой шаг из спеки. `npm run lint-dev` — только если дифф коснулся `packages/**` (чего быть не должно). Скиллы `piece-builder`/`analyze-logs` из эталона не актуальны; `code-review` по диффу задачи — оставить.
8. **ES5/AMD-акцент (новая секция).** Файлы `widget/**` и `bridge/static/widget/**` исполняются загрузчиком amo: AMD-`define`, практический ES5 (без `const`/стрелок/optional chaining — таргет-браузеры клиентов неизвестны), `return` в каждом коллбеке amo, не затирать `window._`/`window.$`/`AMOCRM`, один неймспейс `window.__dzenflow`. Субагентам-кодерам виджета вставлять этот абзац дословно; TypeScript-конвенции моста к виджету не применять.
9. **Security-акцент (новая секция).** Мост — боевой сервис с чужими токенами: для каждого нового эндпоинта в итерации проверить чек-лист — валидация входа по схеме, 403 на неизвестную/revoked связку, CORS-regex, секреты не в логах/ответах ошибок, исходящие только на форк/`*.amocrm.ru`. На V-задачах — мини-обход этого чек-листа по всем эндпоинтам блока (V002/V004 — плюс скилл `security-review`, если диффа набралось много).
10. **Субагенты.** Дефолт sonnet — как в эталоне; но код виджета (ES5/AMD) и всё, что касается JWT/ключей, sonnet-субагентам не делегировать — слишком легко внести современный синтаксис или утечку в лог; им — разведка кода форка, тесты, статика/CSS, i18n.
