# План интеграции приложения в amoCRM

Основание: `RESEARCH.md` + `research/*`. Модель дистрибуции на старте — **приватный виджет для managed-клиентов** (без маркетплейса и модерации), апгрейд до публичного — отдельный этап. Прецеденты боевых виджетов (Sensei/BPMN/Triggeron) показывают: всё задуманное уже делалось.

## Этап 0 — Живая валидация допущений (1–2 дня, dzenteamdev)

1. Скелет приватного виджета (manifest: `settings`, `advanced_settings`, `lcard`, `digital_pipeline`, `salesbot_designer`; фиктивная настройка; zip руками).
2. Проверить: iframe нашего локального превью в `advanced_settings` work-area (CSP — главный риск; прецедент Sensei говорит «пройдёт»); размеры области; `mobile.frame_url`.
3. Прогнать embed-handshake на превью-стенде: signing key → JWT (RS256, `externalProjectId` = account_id) → `configure()` из виджета → конструктор открылся; проверить `allowedEmbedOrigins` с `*.amocrm.ru`-поддоменом клиента.
4. Заодно закрыть UI-QA piece: прокликать дропдауны/refreshers/custom fields/test() в билдере.
   **Выход: снятые открытые вопросы №1–2 из RESEARCH.md, рабочий PoC «amo → iframe → конструктор».**

## Этап 1 — MVP: конструктор внутри amo (managed-путь)

1. **Бэкенд-мост виджета** (отдельный маленький сервис или модуль форка): выдача embed-JWT по install-ключу (онбординг: мы выдаём ключ, клиент вводит в настройках виджета; `onSave` связывает ключ ↔ account_id ↔ project); автодобавление `https://<subdomain>.amocrm.ru` в `allowedEmbedOrigins`; автосоздание amocrm-connection в проекте (наш long-lived токен клиента).
2. **Виджет v1**: `advanced_settings` → страница «Автоматизации» с iframe (SDK: `hideSidebar`, `hideFolders`, `mode` из темы amo, `fontFamily`, locale ru; обработка reload → повторный `configure()`); `lcard` — кнопка «Автоматизации сделки» (пока ведёт на advanced_settings).
3. **Каталог**: ALLOWED-фильтр из аудита (45 pieces, `piecesFilterType`+`piecesTags` в JWT-claims).
4. **«Белый бэкдор»** с первого дня: debug-ключ переопределения `public_path` статики — итерации без перезаливки zip.
   **Выход: клиент открывает наш конструктор из amo, видит только релевантные pieces, flow работают через piece amoCRM.**

## Этап 2 — Автоворонка: DP-шаг и salesbot-шаг

1. Manifest `dp`-блок: поля настройки шага (какой flow запустить, задержка) + `webhook_url` на наш бэкенд-мост (паттерн Triggeron/BPMN); мост маппит account_id+flow → запуск flow (API форка или webhook-триггер piece).
2. `dpSettings`-коллбек: плашка выбора flow (список из нашего API по install-ключу).
3. Manifest `salesbot_designer`-блок (`start_flow` по образцу BPMN `start_bp`) + `onSalesbotDesignerSave`.
   **Выход: «сделка попала в этап → запустился наш flow» настраивается штатной автоворонкой amo.**

## Этап 3 — Карточка сделки

1. `card_sdk`-вкладка: статус автоматизаций сделки (какие flow отработали — из find_events/наших ранов), кнопка ручного запуска.
2. UX-сахар по необходимости: Backbone-модель карточки / MutationObserver (паттерны F5/фильтр-виджета) — только некритичная логика.
3. `mobile.frame_url` — упрощённая мобильная страница.

## Этап 4 — Публичная интеграция (когда managed-поток отлажен)

Техаккаунт → код виджета → OAuth2 вместо install-ключа (redirect на наш бэкенд) → тур-картинки/описания → модерация (закладывать полный ре-ревью на каждое изменение; максимум логики за белым бэкдором) → `widget_page` (полноэкранный пункт меню) вместо advanced_settings как главный вход → «свой каталог внутри одного виджета» (лимит «одна интеграция на вендора»).

## Этап 5 — RU-pieces (параллельная дорожка, ночные циклы ralph)

Приоритет по аудиту: 1) MAX/VK-мессенджеры, 2) телефония (Mango/UIS/Sipuni/Zadarma), 3) платежи (ЮKassa/Robokassa/CloudPayments), 4) Яндекс (Метрика/Директ/Диск), 5) 1С/МойСклад; далее SMS (smsc/sms.ru), Ozon/WB, GetCourse, Tilda, Unisender. Скрыть C-кластер из каталога (это конфиг ALLOWED, не удаление кода). AI: deepseek в ALLOWED сразу; openai/claude — через наш AI-прокси для managed.

## Инфраструктура и эксплуатация (сквозное)

- Хостинг РФ (152-ФЗ): 8 vCPU/16 GB + managed PG (~15 тыс ₽/мес на 50–100 клиентов), ретенция логов ранов с первого дня.
- Rate limits: 7 rps/интеграция и 50 rps/аккаунт (общие с чужими интеграциями клиента) — очереди уже есть (BullMQ).
- Биллинг свой (у amo встроенного нет) — вне этого плана.
- Мониторинг: недоставленные вебхуки, глубина очереди, занятость слотов.

## Зависимости и риски

| Риск | Митигация |
|---|---|
| CSP amo режет iframe (маловероятно — прецедент Sensei) | Этап 0 п.2; fallback — рендер SPA в DOM области (второй режим Sensei) |
| Reload в iframe теряет состояние | Повторный `configure()` в виджете (известное поведение memoryRouter) |
| Модерация публички медленная/капризная | Managed-путь не зависит от неё; белый бэкдор сокращает ресабмиты |
| Piece UI не кликнут человеком | Этап 0 п.4 |
| Ре-пауза движка для wait-механик | Spike T038 ночного цикла (идёт) |
