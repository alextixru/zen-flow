# Поверхности встраивания стороннего приложения в amoCRM/Kommo

Цель: карта всех мест, куда можно встроить наш iframe-конструктор автоматизаций (форк Activepieces, JWT SSO) через JS-виджет amoCRM, с оценкой пригодности каждой поверхности и техническими ограничениями.

Источники: [developers.kommo.com](https://developers.kommo.com) (основная актуальная документация, EN), зеркала `es-/pt-developers.kommo.com`, устаревающий `kommo.com/developers/content/...` (старые статьи, местами дублируют новую доку), локальные боевые манифесты (`reference/triggeron/widget/manifest.json`, `~/motoacademy/amo_comf5_calcfield_48/manifest.json`), конспект курса RocketRed (`~/getcourse/02 Frontend/[2.1]`, `04 Production/[4.1]`, `[4.3]`).

Важно: `amocrm.ru/developers` как отдельное актуальное RU-зеркало **не найдено по рабочему URL** (проверенный путь дал 404) — похоже, RU-документация тоже переехала на `developers.kommo.com` (сайт мультиязычный, ru-локали в доке не увидел, только en/es/pt/tr/id для i18n виджета). Специфики "только для РФ" в найденных материалах не обнаружено — считаю документацию Kommo единым источником для amoCRM.

## 1. Карта поверхностей (locations манифеста)

Все locations задаются в `manifest.json` → `locations: [...]`. Для карточек/списков используется суффикс `-1`/`-0`: `1` — виджет сразу рисует свою панель на карточке, `0` — виджет только инициализируется (и сам решает, показываться ли), это подтверждено и доками, и обоими боевыми манифестами (`lcard-0`, `ccard-0`, `comcard-0`, `cucard-0` у motoacademy; `lcard`, `ccard`, `comcard` без суффикса у triggeron — старый синтаксис, видимо, ещё принимается).

| `lcard` / `ccard` / `comcard` / `cucard` | Карточка сделки / контакта / компании / покупателя. Виджет рисует блок на правой панели фида (или доп. вкладку через Card SDK). | Узкая правая колонка карточки (десктоп), высота — по контенту. | Да, ad-hoc через DOM (`$box.html('<iframe>...')`) — прямого запрета в доке нет, паттерн используется на практике (боевые виджеты грузят статику с внешнего домена `widgets.comf5.ru`). | `render()` (при каждом переходе между разделами/карточками), `init()` (один раз при первой загрузке виджета), `bind_actions()` (по документации должен работать как доп. хук для навешивания событий, **на практике не работает** — вызывается как `init`, один раз — подтверждено на видео курса). | Ничего специфичного к паблику/приватке — работает в обоих типах интеграций. ## 2. Оценка поверхностей под наш iframe-конструктор

**(а) Полноэкранный конструктор.**
Главный кандидат — `widget_page` (свой пункт в левом меню, полноценная страница). Проблема: **требует Public integration**, то есть модерации Kommo Marketplace (см. п.4) — для managed-клиентов на старте это лишний барьер и цикл ревью при каждом изменении.
Практичная альтернатива без паблика — `advanced_settings`: своя страница внутри раздела "Настройки", DOM полностью наш, iframe встраивается свободно, работает и в приватных интеграциях. Минус — находится "внутри" Настроек, а не как отдельный полноценный раздел верхнего уровня, то есть хуже discoverability для конечного пользователя, но для managed-модели (мы сами онбордим клиента и даём прямую ссылку/инструкцию) это не критично.
Рекомендация: на старте — `advanced_settings` как рабочая полноэкранная точка входа в конструктор; `widget_page` — апгрейд после перехода на публичную интеграцию.

**(б) Панель в карточке сделки** ("какие автоматизации активны для этой сделки", кнопка "запустить flow").
`lcard`/`ccard`/`comcard` — прямое попадание. Правая панель узкая, подходит для списка активных flow + кнопки запуска, не для полноценного конструктора. Если нужен более объёмный UI (например, лог запусков) — `card_sdk` даёт целую вкладку вместо узкой полоски, это лучше по эргономике для "панели с историей автоматизаций".

**(в) Шаг в Digital Pipeline** ("при попадании в этап — запустить flow").
Прямое попадание — уже частично реализовано (piece amoCRM с триггерами/экшенами). Технически: форма настроек шага рисуется `dpSettings()`, компактная (не место для конструктора внутри себя — просто выбор, какой flow дёрнуть/какие параметры прокинуть). Основная логика — вебхук `dp.webhook_url`, который **не требует модерации фронтенд-части виджета** (это чисто серверный контракт), что хорошо ложится на наш кейс: пользователь настраивает шаг компактной формой ("выбери flow"), а сам flow редактируется в основном конструкторе (через advanced_settings/widget_page).

**(г) Шаг в Salesbot designer** (`onSalesbotDesignerSave`, `handler_code`/`params`).
Аналогично DP — компактный блок-шаг с формой настроек в визуальном конструкторе бота, возвращающий JSON с exit-кодами для веток сценария. Оба реальных боевых виджета (triggeron, motoacademy) уже используют этот паттерн в проде, значит путь проверенный. Подходит для триггера "запустить flow из сценария бота" аналогично DP, но нативно интегрируется в ветвление бота через `exits`.

**(д) `advanced_settings` как страница настроек подключения.**
Прямое попадание, уже разобрано в (а). Дополнительно годится под классическую задачу "ввести токен/подключить аккаунт", если понадобится нечто отдельное от самого конструктора.

## 3. Технические ограничения встраивания iframe

- **Официального явного заявления "iframe разрешён/запрещён"** в доке Kommo не нашлось — ни подтверждения, ни запрета CSP на `frame-src` внутри widget-контейнеров. Единственное прямое официальное свидетельство того, что iframe — поддерживаемый паттерн: поле `mobile.frame_url` в манифесте (декларативный iframe для мобильной карточки), использующееся в обоих боевых манифестах.
- На практике (боевые виджеты triggeron/motoacademy) статика виджета явно грузится с внешних доменов (`widgets.comf5.ru`, `hooks.widgets.comf5.ru`) — значит, сама amoCRM не режет произвольные внешние ресурсы для установленных виджетов. Это косвенно говорит, что iframe с произвольным `src` (в т.ч. на наш собственный домен-конструктор) должен работать так же, как работает у любого другого установленного виджета — но **это предположение, не проверенный факт**, см. открытые вопросы.
- **HTTPS обязателен** — везде, где документация упоминает `frame_url`/`webhook_url`, это `https://`. Разумно считать обязательным и для src нашего конструктора.
- **RequireJS/AMD-окружение**: `script.js` виджета обязан быть AMD-модулем (`define([...], function(...) {...})`), это подтверждено и доками, и транскриптом курса ("копипаста из доки не работает, если не расставлены `return` в колбэках"). Наш JS-виджет-обёртка должен грузиться и исполняться в этом окружении, но сам контент внутри iframe — уже независимое браузерное окружение, ограничений AMD не имеет.
- **Лимит размера бандла** widget.zip — точной цифры в МБ не нашёл ни в новой, ни в старой доке (только общая рекомендация "не тащите лишние файлы в архив, это тормозит модерацию"). Нужно уточнять на живом стенде/у техподдержки при первой публикации.
- **Мобильное приложение** (`mobile_card`): управляется через `frame_url` — то есть мобильная ветка технически *проще* десктопной (декларативный iframe вместо ручного DOM), но детали общения виджет↔мобильное приложение (postMessage-контракт, доступные методы) в открытой доке не описаны — нужно смотреть на реальном мобильном стенде.
- **Требование "без минификации/обфускации"** для модерации публичных интеграций (`avoids minification and/or obfuscation of files because it complicates and slows down moderation process`, [Public integration checklist](https://developers.kommo.com/docs/getting-listed)) формально противоречит тому, что оба боевых манифеста явно содержат минифицированные бандлы (`reference/README.md` описывает их как "минифицированные", с отдельной "разобранной" копией для анализа). На практике это, видимо, мягкая рекомендация, а не жёсткий блокер — партнёры минифицируют, но должны быть готовы объяснить модератору назначение обфусцированного кода (прямая цитата из курса: *"если оставили `console.log` и его убрал бабель — объясните, это нормально, просто отвечайте на вопросы"*). Не жёсткий технический запрет, скорее операционная процедура ревью.

## 4. Дистрибуция: приватный vs публичный путь

| | Приватная интеграция | Публичная интеграция |
|---|---|---|
| Где работает | Только в том аккаунте amoCRM, где создана | В любом аккаунте, установившем через Marketplace |
| Модерация | **Не требуется** | Обязательна, на **каждое** изменение/версию (два этапа: аудит кода + ручное тестирование живым человеком) |
| Технический аккаунт | Не требуется | Требуется для разработки/тестирования публичных виджетов (создаётся через техподдержку, бесплатный полноценный аккаунт-песочница) |
| OAuth | client_id/secret интеграции, список аккаунтов при авторизации ограничен одним (тем, где создана) | Полноценный OAuth 2.0, пользователь выбирает из своих аккаунтов при установке |
| `widget_page` / `ai_agent` | **Недоступны** (прямое ограничение в доке) | Доступны |
| Стоимость/лимиты установок | Не описаны явно, по факту — не масштабируется за пределы одного аккаунта (это и есть определение приватной интеграции) | Не описаны количественные лимиты |
| Наш кейс (managed-клиенты без маркетплейса) | **Подходит для старта**: заводим виджет в аккаунте каждого клиента как приватную интеграцию (или единую с приваткой на технический аккаунт, если хотим централизованно управлять) | Понадобится, когда захотим self-serve через Marketplace или доступ к `widget_page`/`ai_agent` |

**Минимально для старта (managed-клиенты, наш случай):** приватная интеграция в каждом клиентском аккаунте (или в одном central-аккаунте с раздачей токенов, если архитектура это предполагает) — без модерации, без техаккаунта, без Marketplace-листинга. Ограничения: нет доступа к `widget_page`/`ai_agent`, полноэкранный UI выносим на `advanced_settings`.

Источники: [Kommo for developers](https://developers.kommo.com/docs/kommo-for-developers), [Private integration](https://developers.kommo.com/docs/private-integration), [Getting listed checklist](https://developers.kommo.com/docs/getting-listed), `~/getcourse/04 Production/[4.1] Технические аккаунты amoCRM.md`, `~/getcourse/04 Production/[4.3] Модерация и монетизация.md`.

## 5. Открытые вопросы (проверять на живом стенде dzenteamdev.amocrm.ru)

1. **Реальное поведение CSP/frame-src** для произвольного iframe внутри `lcard`/`advanced_settings`/`widget_page` — задокументированного ответа нет, нужен практический тест: залить тестовый виджет с `<iframe src="https://наш-домен">` внутри DOM-контейнера и проверить браузерную консоль на CSP-блокировки.
2. **Точный размер/пропорции контентной зоны** `widget_page` (`work-area`) и `advanced_settings` — не описаны в пикселях/адаптивности, нужно смотреть вживую (respond to viewport? scrollable?).
3. **Лимит размера widget.zip** в МБ — не найден в открытой доке, уточнять через техподдержку/техаккаунт при подготовке к первой публикации.
4. **Контракт `mobile_card`** (что доступно виджету внутри мобильного iframe, есть ли postMessage API, ограничения по функциональности) — требует теста в реальном мобильном приложении amoCRM.
5. **`ai_agent` location** — коллбэки и формат внедрения не задокументированы подробно (нашли только требование к локализационной строке); если планируем интеграцию с AI-агентами Kommo, нужно отдельное исследование или обращение в поддержку.
6. **`website_chat_button` как location** (в отличие от `crmPluginConfig`-продукта) — не нашли примеров использования и детальных коллбэков, неясно, зачем и как этим пользоваться на практике.
7. **Публичная, но не публикуемая на витрине интеграция** — можно ли пройти путь "Public integration" (ради `widget_page`/`ai_agent`) без цели попасть в общий Marketplace-каталог (т.е. published но unlisted), или публичность = обязательное появление в общем маркетплейсе. Прямого ответа в доке нет.
8. **Точный лимит "Show rows" в `list_selected`** — сколько строк реально обрабатывается в списках (`llist`/`clist`/`tlist`) при массовых операциях — важно для сценариев "запустить flow над выборкой сделок".
