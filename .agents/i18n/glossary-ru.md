# Глоссарий русской локализации Zen-flow

Источник истины для LLM-агентов (драфтеры и судья), допереводящих UI-строки.
Собран из 780 уже переведённых пар. **Приоритет: согласованность с существующим UI.**
Читать в каждом батче. При конфликте между «красивее» и «как в глоссарии» — глоссарий побеждает.
Продукт — форк Activepieces: визуальный билдер автоматизаций (аналог Zapier): сценарии (flows),
интеграции (pieces), запуски, подключения, таблицы, AI-агенты, MCP.

## Тон и стиль

- **Обращение — «вы» со строчной буквы** (не «Вы»). Пример: «убедитесь, что вы скопировали».
- **Без буквы «ё»** — всегда «е»: «включен», «еще», «удален». Существующие переводы её тоже
  не используют; не привносить.
- **Кнопки и действия — инфинитив глагола**: «Опубликовать», «Сохранить», «Удалить», «Создать»,
  «Подключиться», «Импортировать». НЕ существительное («Сохранение» только для процесса-статуса
  «Сохранение...»).
- **Исключения из инфинитива** — служебные кнопки с устоявшейся русской традицией:
  Cancel → «Отмена», Back → «Назад», Done → «Готово», OK → «ОК», Retry → «Повторить».
  НЕ «исправлять» их по общему правилу.
- **Краткость обязательна** для кнопок, пунктов меню и заголовков колонок: русский длиннее
  английского на 15–30%, длинные надписи ломают вёрстку. «Опубликовать», не «Осуществить
  публикацию»; «Настройки», не «Параметры конфигурации». Целиться в ≤1.5 длины оригинала.
- **Капитализация — предложением (sentence case): с заглавной только первое слово** и имена
  собственные. НЕ копировать английский Title Case. Неправильно: «Активные Потоки»,
  «Разблокировать Проекты», «Допустимый Логин Email». Правильно: «Активные потоки»,
  «Разблокировать проекты».
- **Типографика**: кавычки-ёлочки «…» (не "…" и не “…”); тире — «—» (em-dash) между частями фразы,
  дефис «-» только внутри слов. Многоточие троеточием «...» как в оригинале допустимо.
- **«Пожалуйста»** — в новых строках не добавлять без нужды (UI лаконичен). В существующих оставлено —
  не выравнивать массово. Для новых коротких валидаций писать без «пожалуйста».
- Плейсхолдеры-статусы («Загрузка...», «Сохранение...», «Создание...») — существительное + «...».

## Разрешённые несогласованности → канон + запрет

Один термин переведён в данных по-разному. Использовать ТОЛЬКО канон, запрещённые варианты не плодить.

1. **Piece** → канон **интеграция** (решение владельца продукта; старые переводы «часть» при
   доборе НЕ воспроизводить, при правке существующих строк — заменять). Запрещено: «часть»,
   «фигура», «деталь», «запчасть», «Реестр Музея» (NPM Registry ≠ Музей → «Реестр NPM»).
2. **Connection** → канон **подключение**. Запрещено: «соединение».
   (Оба варианта частотны ~50/50; «подключение» ближе привычке CRM-пользователей и совпадает с
   глаголом «подключиться».)
3. **Run / FlowRun** → канон **запуск**. Запрещено: «вызов», «выполненный запуск».
4. **Release** → канон **релиз**. Запрещено: «выпуск».
5. **Flow** → канон **сценарий** (решение владельца продукта: так говорят интеграторы CRM,
   Make/Integromat в ru-версии; старые переводы «поток» при доборе НЕ воспроизводить, при правке
   существующих строк — заменять). Запрещено: «поток», «Flow» латиницей, «динамика»
   («Flows Changes» → «Изменения сценариев», НЕ «Изменения в динамике»), «процесс».
   **Исключение**: OAuth-контексты — «Client Credentials flow» это флоу авторизации, не сценарий
   («авторизация по Client Credentials»).
6. **AI** → в переводимом тексте **ИИ** («ИИ и агенты»); в устоявшихся продуктовых ярлыках
   «AI Credits», «AI Providers», «Text AI» латиница допустима — не переводить их частично.
7. **health** → «состояние» (не «здоровье»): «Trigger Health Status» → «Состояние триггеров».

## Не переводить

**Бренды и продукты**: Zen-flow (наш бренд — НИКОГДА не переводить и не оставлять «Activepieces»;
запрещены кальки «активные образцы/элементы бренда»), Slack, Google, Google Sheets, OpenAI, Git,
SaaS, Node.js, TypeScript, npm/NPM.

**Технические термины (по факту оставлены латиницей в данных)**: API, MCP, JSON, CSV, URL, ID,
IP, SSO, SAML (SAML 2.0), OAuth2, SSH, JS SDK, IDP, `package.json`, `.tgz`, HTTP.
Составные переводятся частично: «API Key» → «Ключ API», «URL перенаправления», «сервер MCP»,
«ключ подписи», но сама аббревиатура остаётся латиницей. Исключение: Webhook → «вебхук»
(кириллицей, см. таблицу).

**Знакомые кальки вместо академизмов** — аудитория (CRM-интеграторы) говорит на языке калек;
академические переводы читаются как машинные: «токен» (НЕ «маркер»), «виджет», «триггер»,
«инстанс», «вебхук», «деплой/развернуть». Selfhosted-словарь: «развернуть» (deploy),
«инстанс» (instance), «переменные окружения» (environment variables) — единообразно с будущей
русской документацией.

**Идентификаторы-примеры** (`org-3412321` и т.п.) — оставить как есть.

## Словарь экосистемы CRM (для pieces и amoCRM-интеграции)

При переводе метаданных pieces, связанных с CRM (и особенно amoCRM), использовать **официальные
русские термины из UI amoCRM** — интегратор мгновенно распознаёт подмену: Lead → «сделка»
(в терминах amoCRM; НЕ «лид», НЕ «дело»), Contact → «контакт», Company → «компания»,
Pipeline → «воронка», Stage/Status → «этап», Note → «примечание», Task → «задача»,
Customer → «покупатель», Custom Field → «поле» / «кастомное поле», Salesbot → «Salesbot»
(не переводить), Digital Pipeline → «Цифровая воронка», Widget → «виджет».

## Правила ICU и плейсхолдеров

- **Переменные в одинарных фигурных скобках `{name}`, `{count}`, `{displayName}` — НЕ переводить,
  НЕ склонять, НЕ трогать пробелы внутри.** Порядок в предложении можно менять под русский синтаксис.
- **Плейсхолдер нельзя просклонять** — фразу строить так, чтобы подставляемое значение стояло
  в именительном падеже, обычно в ёлочках: «Сценарий «{flowName}» удален», «Подключение «{name}»
  создано». НЕ «Удаление {flowName}а», НЕ конструкции, требующие косвенного падежа значения.
- **Разметка** (`**жирный**`, ссылки `[текст](url)`, code-блоки ```` ``` ````, `\n`) — сохранять
  один-в-один; переводить только текст, URL и служебные слова (Setup Instructions и т.п.) не ломать.
- **`#` внутри plural** — это числовое значение, оставлять `#`.
- **Русская плюрализация ОБЯЗАНА иметь формы `one` / `few` / `many` (+ `other`)**, а не только
  `=1`/`other` из английского. Английские `one/other` дают неверные «5 Записи». Схема для целых:
  - `one` — 1, 21, 31… (число оканчивается на 1, кроме 11)
  - `few` — 2–4, 22–24… (оканчивается на 2–4, кроме 12–14)
  - `many` — 0, 5–20, 25–30… (оканчивается на 0, 5–9, или 11–14)
  - `other` — дробные; на практике дублирует `few`.
  - Явные `=0`/`=1` (точные совпадения) можно оставить перед `few/many/other`, если так задумано в UI.

## Примеры плюрализации

Английский `one/other` → русский `one/few/many/other`:

```
en:  {count, plural, one {# item} other {# items}}
ru:  {count, plural, one {# элемент} few {# элемента} many {# элементов} other {# элемента}}
```

Исправление реальной строки `recordsCount` (в данных ошибочно `=0/=1/other`):

```
плохо: {recordsCount, plural, =0 {0 Записей} =1 {1 Запись} other {# Записи}}
верно: {recordsCount, plural, =0 {0 записей} one {# запись} few {# записи} many {# записей} other {# записи}}
```

Дни (число подставляется через `#`):

```
en:  {days, plural, =0 {0 days} =1 {1 day} other {# days}}
ru:  {days, plural, =0 {0 дней} one {# день} few {# дня} many {# дней} other {# дня}}
```

`flowsImported` — в данных строка осталась английской; при доборе перевести с формами:

```
{flowsCount, plural, =0 {Сценарии не импортированы} one {Импортирован # сценарий} few {Импортировано # сценария} many {Импортировано # сценариев} other {Импортировано # сценария}}
```
